# Candidatos de mapeamento por firmware/documentação

Data de consolidação: 2026-07-07

## Critério usado

- `ALTO`: a documentação do projeto indica protocolo/API suficientemente explícito para guiar mapeamento offline por firmware, assets WebUI, ROM pública ou documentação oficial.
- `MÉDIO`: há boa documentação ou evidência comunitária, mas a variação por firmware é relevante e ainda tende a exigir validação viva.
- `BAIXO`: a própria documentação do projeto aponta dependência maior de captura da WebUI real, scraping ou diferenças fortes por firmware de operadora.
- `Firmware público`: link oficial utilizável para baixar firmware/ROM ou central oficial de firmware.
- `Documentação oficial`: link oficial para manuais, support center ou documentação do fabricante quando o firmware não é publicamente exposto.
- `Documentação NetHAL`: ponto interno já mapeado no projeto.

## Leitura objetiva para priorização

- Prioridade natural para mapeamento offline por firmware/documentação: `TP-Link`, `Mercusys`, `Xiaomi`, `ASUS`, `OpenWrt`.
- Melhor caminho por documentação oficial/API, ainda que firmware puro não seja o principal: `MikroTik`.
- Exigem mais cuidado e tendem a pedir captura viva para confirmar payloads reais: `ZTE`, `Intelbras`.
- Não estão bem posicionados, pela documentação atual do projeto, para depender só de firmware baixado: `Huawei`, `FiberHome`, `Nokia`.

## Observações

- Esta tabela não afirma que já exista mapeamento implementado para todas as famílias acima; ela classifica apenas a viabilidade documental de iniciar o mapeamento por firmware público ou documentação oficial.
- Onde a célula de firmware ficou sem link direto, isso significa apenas que nenhum download público oficial foi confirmado nesta rodada; não significa que o fabricante nunca publique firmware por canais restritos.
- Em famílias de CPE de operadora, a documentação do próprio NetHAL continua apontando que o protocolo real costuma depender de firmware customizado, perfis de usuário e reprovisionamento ACS/TR-069.

## Pesquisa comunitária para fabricantes sem hardware físico (2026-07-08)

Levantamento feito para avaliar se existe atalho comunitário (repos open source, engenharia reversa publicada, foruns) que permita avançar o mapeamento de ZTE, Huawei, FiberHome, Nokia e Intelbras sem o equipamento físico em mãos. Conclusão geral: nenhum desses cinco fabricantes tem hoje um atalho específico o bastante para pular captura viva. Os níveis ALTO/MÉDIO/BAIXO da tabela acima não mudam com base nesta pesquisa.

- **ZTE (F689 / F680 / H268A / F6xx / MF266)** — padrão `goform`/cookie `stok` é conhecido pela comunidade, mas os projetos concretos cobrem modems LTE (MF266, MF253M, MF823L, MF286) e routers 5G NG, não as GPON ONTs F680/F689/H268A. Há issues abertas e sem resposta pedindo exatamente isso: [mkst/zte-config-utility#66](https://github.com/mkst/zte-config-utility/issues/66) (F660 v9.0) e [#68](https://github.com/mkst/zte-config-utility/issues/68) (F680). Repos de referência de padrão, não driver pronto: [paulo-correia/ZTE_API_and_Hack](https://github.com/paulo-correia/ZTE_API_and_Hack), [teixeluis/zte-lte-modem](https://github.com/teixeluis/zte-lte-modem).
- **Huawei (HG8145V5 / HG8245Q2 / HG8245H)** — nenhum projeto com payloads reais para esses três modelos. [minzique/huawei-hg8245h5](https://github.com/minzique/huawei-hg8245h5) documenta um modelo diferente (HG8245H5) com só pseudocódigo do hash de login (`SHA256(token)+SHA256(user+token)+SHA256(SHA256(MD5(pwd))+token)`) e endpoints (`/login.cgi`, `/asp/GetRandCount.asp`), sem request/response real. `huawei-lte-api` é para modems LTE, não EchoLife GPON.
- **FiberHome (AN5506 e família AN)** — o achado tecnicamente mais concreto: [gergelykalman.com](https://gergelykalman.com/hacking-isp-cpe-equipment-fiberhome.html) tem endpoints e payloads reais (`/info.asp`, `/rmnt.asp`, `POST /goform/setRemoteControl_3BB`) para o AN5506-02-FG, confirmando backend GoAhead/CGI com padrão `goform`. Também há [bkspc/fiberhomesuperadmin](https://github.com/bkspc/fiberhomesuperadmin) (variante PLDT/Filipinas) e [ndunks/fiberhome-AN5506-04F-](https://github.com/ndunks/fiberhome-AN5506-04F-) (variante Indihome/Indonésia). Todo o material é de firmware customizado por operadora estrangeira, não Brasil (Oi/Vivo) — vale como referência de arquitetura, não como atalho de implementação.
- **Nokia (G-2425G-A/B)** — nada substancial além do já mapeado. [nmikal98/Useful_Links_for_Nokia_G-2425G-A](https://github.com/nmikal98/Useful_Links_for_Nokia_G-2425G-A) é só uma lista de links curada sem conteúdo técnico próprio. O guia do [broadband.forum](https://broadband.forum/threads/nokia-g-2425g-a-unlock-root-step-by-step-detailed-guide.213439/) é sobre root/telnet físico local, não API HTTP para leitura remota.
- **Intelbras (WiFiber / roteadores domésticos)** — nenhum driver ou parser comunitário de leitura encontrado. Único sinal técnico real são dois CVEs que confirmam backend `boaform`/Boa webserver e sessão via cookie, mas são advisories de segurança, não documentação de driver: [GHSA-29f7-g67q-jv68](https://github.com/advisories/GHSA-29f7-g67q-jv68) (WiFiber 120AC inMesh, command injection via `/boaform/formPing6` e `/boaform/formTracert`) e [GHSA-c6f2-5665-5p8p](https://github.com/advisories/GHSA-c6f2-5665-5p8p) (ACtion RF 1200, senha exposta em cookie). Esses CVEs são mais insumo de risco/fingerprint do que atalho de implementação.
- **MikroTik RouterOS** — único caso onde a comunidade agrega valor real: [socialwifi/RouterOS-api](https://github.com/socialwifi/RouterOS-api) (API binária legada, maduro) e [hexatester/rosrestpy](https://github.com/hexatester/rosrestpy) (REST v7). São wrappers maduros sobre API já oficial, não engenharia reversa de protocolo fechado — reforça reuso de lib em vez de reimplementar cliente do zero, sem mudar o nível ALTO já atribuído.

Releitura: FiberHome e ZTE têm o padrão arquitetural mais bem descrito publicamente (`goform`/CGI, cookie `stok`), então valem revisita se aparecer hardware físico ou firmware brasileiro real. Os demais permanecem bloqueados em captura viva.
